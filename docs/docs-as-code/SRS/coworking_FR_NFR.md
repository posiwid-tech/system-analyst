Файл описывает требования к разрабатываемой системе по кейсу 2: Проектирование сервиса бронирования рабочих пространств

## 1. Функциональные требования
Предусловие для всех функциональных требований: 
1. Пользователь зарегистрирован в системе.
2. Пользователь аутентифицирован в системе.

### 1.1. Просмотр площадок и рабочих пространств

**FR-01. Просмотр списка площадок**
Система должна предоставлять пользователю возможность получить список доступных для бронирования площадок.

**FR-02. Просмотр информации о площадке**
Система должна предоставлять пользователю информацию о выбранной площадке.

**FR-03. Просмотр рабочих пространств площадки**
Система должна предоставлять пользователю список рабочих пространств, относящихся к выбранной площадке.

**FR-04. Просмотр информации о рабочем пространстве**
Система должна предоставлять пользователю информацию о выбранном рабочем пространстве, включая его тип и стоимость аренды.

### 1.2. Просмотр доступности

**FR-05. Запрос доступных временных слотов**
Система должна предоставлять пользователю возможность запросить доступные временные слоты для выбранного рабочего пространства за указанный период времени.

**FR-06. Отображение только доступных слотов**
Система должна возвращать в результатах поиска только временные слоты, доступные для бронирования на момент обработки запроса.

**FR-07. Отображение стоимости слота**
Система должна предоставлять пользователю стоимость бронирования для каждого доступного временного слота.

### 1.3. Создание бронирования

**FR-08. Создание бронирования**
Система должна предоставлять авторизованному пользователю возможность создать бронирование выбранного рабочего пространства на выбранный временной слот.

**FR-09. Проверка существования рабочего пространства**
Перед созданием бронирования система должна проверить статус доступности выбранного рабочего пространства.

**FR-10. Проверка доступности слота**
Перед созданием бронирования система должна проверить доступность выбранного рабочего пространства на указанный временной интервал.

**FR-11. Предотвращение двойного бронирования**
Система не должна создавать новое бронирование рабочего пространства, если его временной интервал пересекается с временным интервалом другого действующего бронирования этого рабочего пространства.

**FR-12. Фиксация стоимости бронирования**
При создании бронирования система должна сохранить стоимость аренды, действующую для выбранного рабочего пространства и временного слота на момент создания бронирования.

**FR-13. Создание идентификатора бронирования**
При успешном создании бронирования система должна присвоить бронированию уникальный идентификатор.

**FR-14. Сохранение бронирования**
При успешном создании бронирования система должна сохранить информацию о бронировании в базе данных.

**FR-15. Подтверждение успешного бронирования**
После успешного создания бронирования система должна предоставить пользователю подтверждение создания бронирования с указанием идентификатора бронирования и его параметров.

**FR-16. Отказ при недоступности слота**
Если выбранный временной слот недоступен на момент создания бронирования, система должна отказать в создании бронирования и сообщить пользователю о недоступности выбранного слота.

### 1.4. Получение информации о бронировании

**FR-17. Получение бронирования по идентификатору**
Система должна предоставлять пользователю возможность получить информацию о принадлежащем ему бронировании по идентификатору бронирования.

**FR-18. Состав информации о бронировании**
Система должна предоставлять для бронирования идентификатор бронирования, площадку, рабочее пространство, время начала, время окончания, стоимость и текущий статус бронирования.

**FR-19. Ограничение доступа к бронированию**
Система не должна предоставлять пользователю информацию о бронировании, принадлежащем другому пользователю.

### 1.5. Отмена бронирования

**FR-20. Отмена бронирования пользователем**
Система должна предоставлять пользователю возможность отменить принадлежащее ему действующее бронирование.

**FR-21. Изменение статуса при отмене**
При успешной отмене система должна изменить статус бронирования на `CANCELLED`.

**FR-22. Освобождение временного интервала**
После успешной отмены бронирования соответствующий временной интервал рабочего пространства должен стать доступным для нового бронирования, если отсутствуют другие ограничения доступности.

**FR-23. Запрет повторной отмены**
Система не должна изменять состояние бронирования при повторном запросе на отмену уже отменённого бронирования.

**FR-24. Запрет отмены чужого бронирования**
Система не должна позволять пользователю отменять бронирование, принадлежащее другому пользователю.


## 2. Нефункциональные требования

### 2.1. Производительность

**NFR-01. Время получения доступных слотов**
Система должна обрабатывать не менее 95 % запросов на получение доступных временных слотов не более чем за 2 секунды при штатной нагрузке.

**NFR-02. Время создания бронирования**
Система должна обрабатывать не менее 95 % запросов на создание бронирования не более чем за 3 секунды при штатной нагрузке.

**NFR-03. Время получения бронирования**
Система должна обрабатывать не менее 95 % запросов на получение информации о бронировании не более чем за 1 секунду при штатной нагрузке.

### 2.2. Масштабируемость

**NFR-04. Горизонтальное масштабирование**
Система должна поддерживать горизонтальное масштабирование компонентов, обрабатывающих пользовательские запросы, без изменения клиентского API.

**NFR-05. Поддержка нескольких площадок**
Система должна обеспечивать одновременную работу не менее чем с 1 000 площадок без изменения архитектуры пользовательского взаимодействия.

### 2.3. Доступность и отказоустойчивость

**NFR-06. Доступность системы**
Доступность основных функций системы должна составлять не менее 99,9 % в месяц, за исключением согласованных периодов технического обслуживания.

**NFR-07. Сохранность подтверждённых бронирований**
Отказ отдельного экземпляра прикладного сервиса не должен приводить к потере ранее подтверждённых бронирований.

### 2.4. Консистентность данных

**NFR-08. Уникальность активного бронирования временного интервала**
Система должна гарантировать невозможность создания двух действующих бронирований одного рабочего пространства на пересекающиеся временные интервалы, включая обработку параллельных запросов.

**NFR-09. Атомарность создания бронирования**
Операция создания бронирования должна выполняться атомарно: при неуспешном завершении операции в системе не должно оставаться частично созданного бронирования.

**NFR-10. Атомарность отмены бронирования**
Операция изменения статуса бронирования при отмене должна выполняться атомарно.

### 2.5. Безопасность

**NFR-11. Аутентификация пользователей**
Доступ к операциям создания, просмотра и отмены пользовательских бронирований должен предоставляться только аутентифицированным пользователям.

**NFR-12. Авторизация пользователей**
Система должна проверять полномочния пользователя на просмотр или изменение бронирования перед выполнением соответствующей операции.

**NFR-13. Защита передаваемых данных**
Взаимодействие клиентов с системой через публичный API должно осуществляться по защищённому протоколу HTTPS с использованием TLS версии 1.2 или выше.

### 2.6. Надёжность обработки запросов

**NFR-14. Идемпотентность создания бронирования**
Повторная отправка одного и того же запроса на создание бронирования с одинаковым ключом идемпотентности не должна приводить к созданию нескольких бронирований.

**NFR-15. Идемпотентность отмены бронирования**
Повторная отправка запроса на отмену одного и того же бронирования не должна приводить к ошибочному изменению данных бронирования.

### 2.7. Аудит и наблюдаемость

**NFR-16. Журналирование операций с бронированиями**
Система должна журналировать операции создания и отмены бронирований с указанием идентификатора бронирования, типа операции и времени выполнения операции.

**NFR-17. Мониторинг ошибок**
Система должна регистрировать технические ошибки, возникающие при обработке запросов создания, получения и отмены бронирований.

**NFR-18. Метрики системы**
Система должна предоставлять технические метрики, позволяющие контролировать количество запросов, время их обработки и количество ошибок.
